## КриптоПРО

*CSP* — Cryptographic Service Provider — «поставщик криптографических услуг»

*СКЗИ* — средство криптографической защиты информации

*КРИПТО-ПРО* — это компания, которая создает средства криптографической защиты информации (СКЗИ).
Самый известный их продукт — *КриптоПро CSP* — это программа, которая позволяет использовать электронную
подпись на компьютере. Они не выдают сертификаты сами, а предоставляют для этого инструменты.

Другие СКЗИ: Рутокен, Инфокрипт

## Определения

*Закрытый ключ*:
- храниться только у тебя
- можно подписать документ, расшифровать

*Открытый ключ*:
- можно раздавать всем
- можно проверить подпись

*Сертификат*
- `.cer` или `.crt`
- бинарник
- хранит открытый ключ и метаданные (ФИО владельца, срок, кем выдан, серийный номер)

Пара ключей создаются вместе и связана математически.
Голый ключ хранить обычно плохо, поэтому его упаковывают в контейнер.

*Контейнер*:
- `.key`
- бинарник (часто зашифрованный)
- содержит закрытый ключ, маски ключей, заголовки, пароли, служебную инфу
- программный контейнер находится в папке `/var/opt/cprocsp/keys/` или на флешке
- аппаратный контейнер находится на рутокене, JaCarta, eSmart и т.д.

Контейнер может стоять из нескольких файлов .key:
- header.key,    Заголовок контейнера, имя владельца, срок действия и т.д.
- masks.key,     Первая часть масок ГОСТ (защита от извлечения)
- masks2.key,    Вторая часть масок
- primary.key,   Основная часть зашифрованного закрытого ключа
- primary2.key,  Дублирующая часть (для надёжности)
- имя.000,       Папка-контейнер (внутри служебные данные)

*УЦ* — удостоверяющий центр. Организация, которой государство официально разрешило выдавать электронные подписи (УЦ ФНС, контур, таксом).
Приходишь к ним с паспортом, они проверяют и выдают тебе электронную подпись.
Выданный сертификат подписывается своим закрытым ключом.

*CA* (Certificate Authority) — удостоверяющий центр

*Корневой сертификат*
- выпускают минцифры РФ, подписывают ими сертификаты УЦ
- храниться в `/var/opt/cprocsp/certs/uroot/` или `/etc/opt/cprocsp/certs/`

*Цепочка доверия*: мой сертификат подписан УЦ, а их сертификат корневым сертификатом

*Отпечаток* (thumbprint) — уникальный идентификатор сертификата, SHA-1 или SHA-256 хеш от всего содержимого сертификата. Нужен чтобы идентифицировать сертификат.

*УКЭП* (Усиленная квалифицированная электронная подпись)
УКЭП жестко привязана к конкретному человеку через квалифицированный сертификат, выпущенный аккредитованным центром.
Ведается только сертифицированными ФСБ СКЗИ

*УНЭП* (Усиленная неквалифицированная электронная подпись)
УНЭП может существовать без такого сертификата, подтверждая личность через подтвержденную учетную запись на "Госуслугах" (ЕСИА)
Может использовать любые СКЗИ, включая open-source решения

*Алгоритм подписи* (пример:  ГОСТ Р 34.10-2012)

*Алгоритм хеширования* (пример: ГОСТ Р 34.11-2012) — это математическое преобразование, которое принимает на вход файл (сообщение)
произвольного размера и возвращает на выходе строку фиксированной длины, называемую хэш-суммой. Не использует ни закрытый, ни открытый ключи.

## PEM
Файлы в формате PEM (*Privacy-Enhanced Mail*) — это текстовые файлы, которые используются для хранения и
передачи различных криптографических объектов, таких как ключи и сертификаты.

Проще говоря, это стандартный "конверт" для безопасных данных в интернете.
PEM-файл легко узнать по его структуре. Это обычный текстовый файл, который содержит три обязательные части:
1) Заголовок: Строка, которая всегда начинается с -----BEGIN и заканчивается -----. В ней указан тип хранящихся данных (например, CERTIFICATE или PRIVATE KEY)
2) Тело: Сами данные, закодированные в формате Base64. Это делает двоичные данные удобочитаемыми и позволяет передавать их по текстовым протоколам 
3) Нижний колонтитул: Строка, похожая на заголовок, которая начинается с -----END и также содержит тип данных

## CLI КриптоПРО
*csptest* — работа с контейнерами и ключами (перечислить, проверить, сгенерировать, удалить)
*certmgr* — работа с сертификатами и хранилищами (установить, удалить, посмотреть)
*cryptcp* — подписать/зашифровать файл из командной строки (платная часть)
*cpverify* — проверить подпись на файле
*cpconfig*


## Подписание
```sh
/opt/cprocsp/bin/amd64/cryptcp -sign -detached -strict -nochain -der -thumbprint $THUMBPRINT -pin $PIN ./documentPath ./pathSign
```

1) Программа считает хеш документа
2) КриптоПро берет закрытый ключ из контейнера
3) Делает подпись (из хеша и закрытого ключа)
4) (опционально) К документу приклеивается подпись + сертификат


## Проверка
```sh
/opt/cprocsp/bin/amd64/cryptcp -verify ./documentPath -detached -nochain -thumbprint $THUMBPRINT ./pathSign
```

1) Получатель берет твою подпись, расшифровывает открытым ключом
2) Получает хэш файла, считает хэш документа
3) Если он совпали, значит документ не изменён


## Выдача сертификата
*CSR* — запрос на выдачу сертификата, содержит мета данные (ФИО, организацию и т. д.) + открытый ключ + подпись этого запроса закрытым ключом
CSR отправляется в УЦ, он выдаёт обратно сертификат

CSR бывает 2 видов:
- PKCS #10 (формат запроса на сертификат, имеет расширение `.p10` или `.csr` или `.req`)
- PKCS #7 (имеет расширение `p7c` или `.p7b`)

*PKCS* (Public-Key Cryptography Standards) — стандарты криптографии с открытым ключом


### Создание запроса
```sh
cryptcp -creatrqst -dn "CN=Иванов Иван Иванович, SNILS=123-456-789 01" -cont "\\.\HDIMAGE\abcd1234.000" request.p10
```

1) Генерируется новая пара ключей (закрытый + открытый) непосредственно в момент создания запроса.
2) Открытый ключ встраивается в CSR.
3) Закрытым ключом (тем самым, который только что сгенерирован) подписывается сам CSR.
...
4) Полученный CSR отправляется в удостоверяющий центр
5) УЦ обрабатывает запрос и выдаёт обратно сертификат


## OID

*OID* (Object Identifier) — это уникальный иерархический числовой идентификатор, который используется для определения:
- алгоритмов (как подписан сертификат, какой алгоритм хэширования использован)
- политик (для каких целей выпущен сертификат)
- расширений (дополнительных полей в сертификате)
- атрибутов субъекта (полей в отличительном имени — Distinguished Name)

### Атрибуты Subject (DN)
*1.2.643.3.131.1.1* (INN) — ИНН физ лица
*1.2.643.100.1* (OGRN) — ОГРН
*1.2.643.100.3* (SNILS) — СНИЛС физ лица
*1.2.643.100.4* (INNLE) — ИНН юр лица
*1.2.643.100.5* (OGRNIP) — ОГРНИП
*.2.840.113549.1.9.1* (E) — адрес электронной почты

*2.5.4.3* (commonName, CN) — имя субъекта: для сайта — домен, для человека — ФИО
*2.5.4.4* (SN) — фамилия сотрудника
*2.5.4.6* (countryName, C) — код страны (две буквы, например, RU)
*2.5.4.8* (stateOrProvinceName, ST) — область или регион
*2.5.4.10* (organizationName, O) — название организации
*2.5.4.11* (organizationalUnitName, OU) — название подразделения
*2.5.4.12* (T) — должность
*2.5.4.42* (GN) — имя и отчество сотрудника


### Алгоритмы
*1.2.840.113549.1.1.1* (rsaEncryption) — алгоритм RSA для шифрования и подписи
*1.2.840.113549.1.1.11* (sha256WithRSAEncryption) — подпись RSA с хэшем SHA-256
*1.2.840.10045.2.1* (ecPublicKey) — алгоритм ECC (эллиптическая криптография)
*1.2.840.10045.4.3.2* (ecdsa-with-SHA256) — подпись ECDSA с хэшем SHA-256

### Key Usage (KU)
*2.5.29.15* (keyUsage) — битовая маска, определяющая основные операции с ключом (цифровая подпись, шифрование ключа, подпись сертификатов и т.д.)

### Extended Key Usage (EKU)
*1.2.643.100.111* (Subject Signing Tool) — название СКЗИ с помощью которого создан запрос на сертификат
*1.3.6.1.5.5.7.3.1* (serverAuth) — TLS-сертификат для аутентификации сервера (веб-сайт)
*1.3.6.1.5.5.7.3.2* (clientAuth) — TLS-сертификат для аутентификации клиента (пользователь, устройство)
*1.3.6.1.5.5.7.3.3* (codeSigning) — подпись программного кода
*1.3.6.1.5.5.7.3.4* (emailProtection) — защита электронной почты (S/MIME)
*1.3.6.1.5.5.7.3.8* (timeStamping) — постановка метки времени (Timestamp Authority)
*1.3.6.1.4.1.311.10.3.12* — подпись драйверов Windows (Microsoft)

### Базовые расширения сертификата
*2.5.29.15* (keyUsage) — ограничивает допустимые операции с открытым ключом (см. выше)
*2.5.29.19* (basicConstraints) — определяет, является ли сертификат CA (удостоверяющим центром) и задает глубину цепочки подчинения
*2.5.29.37* (extKeyUsage) — контейнер для одного или нескольких EKU, перечисленных выше

### Расширения для построения цепочек доверия
*2.5.29.14* (subjectKeyIdentifier, SKI) — хэш открытого ключа субъекта, используется для связывания сертификатов в цепочке
*2.5.29.35* (authorityKeyIdentifier, AKI) — хэш открытого ключа издателя (CA), указывает, какой CA выпустил сертификат

### Расширения для идентификации
*2.5.29.17* (subjectAltName, SAN) — альтернативные имена субъекта: домены, IP-адреса, email-адреса (современная замена CN для мультидоменных сертификатов)
*1.2.840.113549.1.9.1* (emailAddress) — email-адрес (устаревший атрибут, рекомендуется использовать SAN)

### Политики
*2.5.29.32* (certificatePolicies) — политики сертификата, определяющие, по каким правилам выпущен сертификат (например, OID политики CA, указание на квалифицированный сертификат)

## Установка сертификата ЭП

1.
Скопировать закрытый ключ в хранилище, где <username> - имя пользователя linux:
sudo cp -R /path/to/key/999996.000 /var/opt/cprocsp/keys/<username>/
--------

2.
Поставить «минимальные» права:
sudo chmod 600 -R /var/opt/cprocsp/keys/<username>/999996.000/
--------

3.
Узнать реальное название контейнера:
sudo /opt/cprocsp/bin/amd64/csptest -keyset -enum_cont -verifycontext -fqcn
sudo /opt/cprocsp/bin/amd64/csptest -keyset -enum_cont -fqcn -verifyc | iconv -f cp1251  
--------

4.
Ассоциировать сертификат с контейнером, сертификат попадет в пользовательское хранилище My:
sudo /opt/cprocsp/bin/amd64/certmgr -inst -file "/home/vit sol open.cer" -cont '\\.\HDIMAGE\OOO_VDELO_Solovev_28022023_1677558645 копия'

--------

5.
--------

5.
Установить сертификат УЦ из-под пользователя root командой:
sudo /opt/cprocsp/bin/amd64/certmgr -inst -store uroot -file /path/to/file/CA.cer
--------

6.
Проверка успешности установки закрытого ключа
sudo /opt/cprocsp/bin/amd64/certmgr --list
--------

Если выводится PrivateKey Link: Yes. Container: HDIMAGE\\999996.000\D7BB, то есть и сертификат, и закрытый ключ, а если выводится PrivateKey Link: No - связи нет, и использовать такой контейнер для подписи не удастся.
--------
